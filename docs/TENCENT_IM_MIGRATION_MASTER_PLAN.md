# Raver 腾讯云 IM 迁移总方案

> 状态：进行中  
> 当前分支：`thirdPartyIM`  
> 文档创建时间：2026-04-27 22:59:06 +0800  
> 目标：彻底放弃当前自建 / OpenIM 路线，切换到腾讯云 IM 作为唯一聊天基础设施，并把全过程拆成可执行、可跟踪、可交接的步骤。

## 1. 结论先行

本次迁移建议采用下面这条路线：

```text
Raver 保留业务主系统
腾讯云 IM 负责聊天基础能力
当前 OpenIM 代码与实验资产不再继续投入
旧自建消息模型不再继续演进
迁移采用“硬切换 + 只读归档”，不做长期双写
```

核心原因：

- 当前仓库已经存在较重的 OpenIM 代码面和实验目录，但你已经明确要放弃这条路线。
- Raver 现有业务系统已经有用户、关系、小队、通知、活动等主数据，最适合继续做业务权威源。
- 腾讯云 IM 当前官方能力可以覆盖 `SDKAppID + UserID + UserSig` 登录、C2C 单聊、群组、离线推送、回调、消息自定义扩展等关键需求。
- 现有 iOS 聊天页已经有自定义 UI 和业务装配逻辑，最适合接入腾讯云 IM SDK 能力层，而不是整套推倒重做成另一个第三方 UI 工程。

## 3. 目标架构

### 3.1 最终架构

```text
iOS App
  -> 登录 Raver
  -> 向 Raver BFF 获取腾讯云 IM 登录参数
  -> iOS 使用腾讯云 IM SDK 登录、拉会话、收发消息

Raver Server
  -> 用户、关系、小队、活动、权限、通知主库
  -> 生成 UserSig
  -> 调腾讯云 IM REST API 管理用户与群组
  -> 接收腾讯云 IM 回调
  -> 处理业务镜像、审核、风控、运营后台

腾讯云 IM
  -> 单聊
  -> 群聊
  -> 会话与未读
  -> 离线消息
  -> 推送与回调
  -> 自定义消息
```

### 3.2 职责划分

Raver 继续负责：

- 用户注册、登录、封禁、注销
- 用户资料、好友关系、关注关系
- Squad / Event / Notification 等业务主数据
- 群成员权限权威源
- 审核策略、风控策略、后台运营
- IM 登录票据发放接口

腾讯云 IM 负责：

- C2C 单聊
- 群组会话
- 消息存储与同步
- 未读数
- 自定义消息投递
- 离线推送能力
- 会话事件回调

## 4. 迁移原则

### 4.1 原则一：完全放弃 OpenIM 继续研发

- 不再新增任何 OpenIM 功能。
- 不再修复任何 OpenIM 业务问题，除非只是为了保证切换窗口期可用。
- OpenIM 只允许作为“参考旧实现”存在于历史提交或短期只读代码中。

### 4.2 原则二：不做长期双栈

- 不建议长期维护 `OpenIM + 腾讯云 IM` 双实现。
- 允许极短期并行窗口，仅用于验证新链路。
- 并行窗口结束后，旧 OpenIM 代码必须进入删除清单。

### 4.2.1 强制执行补充（2026-04-28 新增）

- 后续所有改动默认遵循：**只保留腾讯云 IM 新逻辑**。
- 不做新旧协议兼容分支，不保留 OpenIM 兜底路径，不新增“回退到 OpenIM”开关。
- 若出现“为了兼容旧协议/旧字段/旧控制器”的需求，默认判定为不进入主线，需单独评审后才允许临时落地。

### 4.3 原则三：消息数据默认不迁移，先归档

默认建议：

- 旧自建 IM 数据先做数据库快照或导出归档。
- 不把 OpenIM 实验消息历史迁入腾讯云 IM。
- 如果业务后续明确要求“保留老聊天记录”，再单独立项做历史导入。

这样做的好处：

- 工程复杂度低很多。
- 可以尽快把产品带到可运营状态。
- 避免把 OpenIM 的历史兼容包袱带进新方案。

## 6. 腾讯云 IM 方案选择

### 6.1 为什么选腾讯云 IM SDK，不直接上整套 TUIKit UI

推荐生产方案：

```text
接腾讯云 IM SDK 能力层
保留 Raver 当前聊天 UI 与业务路由层
```

原因：

- 当前 iOS 已经有自定义消息模块、会话装配和页面协调器。
- 直接用第三方整套 UI 重新盖住现有消息页，会把导航、样式、业务卡片、小队入口、设置页全部重做一遍。
- 用 SDK 能力层接入，未来更容易保留 Raver 自己的交互风格。

TUIKit 的合适用途：

- 2 天内快速做一个 PoC。
- 验证腾讯云 IM 环境、UserSig、基础收发是否通。
- 不建议作为正式产线实现的主路线，除非你决定放弃现有聊天 UI 并接受整体重写。

### 6.5 iOS 交互逻辑对齐原则（新增）

为了加快落地并降低学习成本，一期新增一个明确原则：

- 聊天相关界面的“操作逻辑”尽量与腾讯云 iOS UIKit 对齐。
- 聊天相关界面的“视觉效果”也可按腾讯云 iOS UIKit 的信息层级和操作布局对齐。
- 品牌色与基础字体沿用 Raver 主题，不做整套 TUIKit 页面替换。

一期优先对齐的操作逻辑：

- 会话列表：进入、返回、未读更新、置顶/免打扰/删除等基础会话行为。
- 聊天页：文本发送、图片/视频发送、失败态、长按菜单动作顺序。
- 消息能力：引用回复、`@`、已读回执、撤回后的占位展示。
- 群会话：成员管理入口、群公告入口、群资料入口的操作路径。

### 6.2 腾讯云 IM 中的关键概念

- `SDKAppID`：应用标识。
- `UserID`：IM 用户唯一标识。
- `UserSig`：登录鉴权签名。
- `C2C`：1v1 单聊。
- `Group`：群聊。
- `Custom Message`：自定义消息，适合活动卡片、DJ 卡片、系统卡片。

### 6.3 用户与群组映射建议

建议使用稳定可逆映射，但必须满足腾讯云 IM 官方 `UserID` 的长度与字符限制：

```text
Raver User.id   -> TIM userID
Raver Squad.id  -> TIM groupID
Raver Event.id  -> TIM event groupID
```

建议规则：

- 不直接使用完整 UUID 拼接前缀作为 `UserID`
- `UserID` 需要控制在腾讯云 IM 官方要求范围内
- 推荐使用稳定短 ID 映射表，例如：
  `tu_{short_id}`
- `Squad groupID` 推荐使用：
  `sg_{short_id}`
- `Event groupID` 推荐使用：
  `eg_{short_id}`

要求：

- 只使用稳定主键，不使用昵称、手机号、用户名。
- 在业务表中保留映射字段，方便定位问题。
- 所有映射逻辑收口到一个服务中，避免前后端各自拼接。

### 6.4 群类型建议

基于腾讯云 IM 官方当前群组类型说明，以及你当前已经明确的小队管理员、群管理、50-100 人规模需求，建议：

- Squad 群：优先用 `Public`
- 活动临时群：优先用 `Meeting`
- 超大规模活动群：如确有需要，再评估 `Community` 或 `AVChatRoom`

原因：

- `Public` 更适合带管理员、入群审核、稳定成员管理的正式业务群。
- `Meeting` 更适合临时进入退出的活动场景。
- `Community` 和 `AVChatRoom` 适合大规模群体，但会引入额外的套餐、能力和运营复杂度。
- `Work` 群虽然轻量，但不适合你当前已经明确的一期群管理诉求。
