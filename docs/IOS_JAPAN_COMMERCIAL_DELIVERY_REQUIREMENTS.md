# RaveHub 日本 iOS 商用级交付需求规格

> 版本：2026-05-15  
> 适用范围：iOS App、Widget、Notification Service、BFF/API、管理后台、公开 Web 法务页面、运营流程。  
> 原则：不按“先 MVP、后补齐”拆口径。本文定义的是商用完整交付标准；开发可分批，但最终验收必须覆盖全链路。

## 0. 总体交付原则

### 0.1 商用级定义

一个功能只有在以下链路完整时才算完成：

- 用户端：入口清晰、状态可理解、错误可恢复、权限拒绝有替代路径。
- 服务端：数据模型、API、鉴权、幂等、状态机、审计、通知、任务重试完整。
- 管理后台：运营人员可查询、处理、撤销、备注、导出、追踪责任人。
- 风控合规：最小化收集、隐私说明、用户申诉、数据保留、日志脱敏。
- 国际化：中文、英文、日文均可用，且法务、权限、通知、邮件/站内信都覆盖。
- 验证：单元/集成/端到端/回归清单覆盖核心路径和异常路径。
- 可运营：指标、SLA、告警、报表、客服话术、Review Notes 资料齐备。

### 0.1.1 需求追踪方式

本文定义“做到什么程度才算商用可交付”。实际开发进度统一记录在 `docs/IOS_JAPAN_APP_STORE_COMPLIANCE_PROGRESS.md`，每个功能必须拆成可勾选的全链路 checkbox。任何条目只有同时满足以下条件才允许从 `[ ]` 改为 `[x]`：

- App、API、后台、数据、通知、审计、法务/文案、测试均覆盖；若某一端不适用，必须在进度文档写明原因。
- 中文、英文、日文文案均完成，且日文不是机器直译未审状态。
- 有失败路径和边界状态，例如权限拒绝、封禁到期、重复举报、申诉驳回、外部系统重试失败。
- 有可复测的验收证据，例如命令、截图、后台记录、API 响应、测试账号路径或 Review Notes 文案。
- 运营/客服能处理后续生命周期：查询、导出、恢复、撤销、备注、通知用户。

### 0.2 全局角色

- 游客：未登录浏览用户。
- 普通用户：已注册/登录用户。
- 被限制用户：部分能力受限，例如不能发帖或私信。
- 临时封禁用户：指定时间内不可登录或不可使用核心互动能力。
- 永久封禁用户：账号永久不可使用，除申诉恢复外不可登录。
- 管理员：可处理举报、内容、账号处罚、法务配置。
- 审核员/运营：可处理举报和内容审核，但不能修改系统配置。
- 超级管理员：可处理高风险动作，例如永久封禁、批量处理、删除数据。

### 0.3 全局审计要求

所有高风险动作必须写审计：

- 谁操作：用户 ID、管理员 ID、角色、IP、User-Agent。
- 操作什么：目标类型、目标 ID、前后状态。
- 为什么操作：原因码、备注、证据。
- 何时生效：创建时间、生效时间、到期时间、撤销时间。
- 可追溯：关联举报、申诉、工单、通知、自动规则版本。

## 1. 中英日三语适配

### 1.1 目标

现有 App 支持中文和英文，需要扩展到日文，并达到日本区商用发布标准。三语不是简单替换文案，而是 UI、内容、法务、通知、后台、App Store 元数据和运营话术的完整适配。

### 1.2 语言范围

- App UI：登录、注册、设置、Feed、消息、发现、活动、DJ、Set、小队、通知、错误提示、空状态。
- 系统权限文案：定位、相机、麦克风、相册、推送软提示。
- 用户可见内容状态：审核中、已拒绝、已封禁、申诉中、已恢复。
- 法务页面：隐私政策、服务条款、社区规范、特商法页面、数据请求、版权投诉。
- 通知：站内信、推送标题/正文、邮件/SMS 模板。
- 管理后台：菜单、筛选、状态、处理原因、导出字段、审核操作。
- App Store Connect：名称、副标题、描述、关键词、隐私 URL、客服 URL、审核说明日文版。

### 1.3 App 端需求

- `AppLanguage` 增加 `ja`，支持系统语言自动识别 `ja-*`。
- 语言设置页展示：跟随系统、中文、English、日本語。
- 所有硬编码 `L(zh,en)`/`LL(zh)` 文案升级为三语能力。建议建立统一 `LocalizedStringKey`/资源表，不继续无限扩展散落字典。
- 所有用户可见日期、时间、货币、数字使用 locale 格式化：
  - 中文：`zh-Hans`
  - 英文：`en`
  - 日文：`ja-JP`
- 内容字段支持多语言：
  - 活动：名称、描述、地点、票务说明。
  - DJ/厂牌/资讯/学习内容：标题、简介、正文。
  - 审核原因：原因码固定，展示文案本地化。
- 用户生成内容不强制机器翻译；但系统标签、状态、按钮必须三语。
- 所有 UI 需要验证日文长词、假名混排、换行、按钮宽度、动态字体。

### 1.4 后端与内容模型

- 所有系统枚举使用稳定 reason code，不把中文文案写进业务字段。
- API 返回本地化结构：
  - `titleI18n: { zh, en, ja }`
  - `bodyI18n: { zh, en, ja }`
  - 保留 fallback 规则：`ja -> en -> zh -> raw`。
- 通知模板表按 locale 存储，至少 `zh-CN`、`en`、`ja-JP`。
- CMS/后台录入必须支持三语字段和缺失检查。
- 搜索支持日文假名/罗马字/英文名/中文名的基础匹配；关键内容建立 alias。

### 1.5 管理后台

- 后台可筛选“缺日文文案”的内容。
- 法务页面和通知模板支持版本发布、预览、回滚。
- 运营可导出缺失翻译清单。
- 任何自动翻译结果必须有人工确认状态，避免把未确认翻译直接发布到法务/处罚通知。

### 1.6 测试与验收

- 三语截图回归：登录、设置、举报、封禁提示、账号删除、法务页面、活动详情、消息页。
- 日文系统环境冷启动后默认进入日文。
- 日文权限弹窗、推送软提示、封禁通知、账号删除说明均可读。
- App Store 元数据日文完整，无中文残留在日本区首屏关键路径。

### 1.7 商用交付 checklist

- [ ] App 语言模型支持 `zh-Hans`、`en`、`ja-JP` 和跟随系统。
- [ ] App 所有用户可见文案移出散落硬编码，建立可审计的三语资源或统一本地化层。
- [ ] Web 法务页、管理后台、通知模板、邮件/站内信均有三语版本。
- [ ] 后端系统枚举只保存 reason code，不保存单一语言业务文案。
- [ ] CMS/后台支持三语内容录入、缺失检查、预览、发布、回滚。
- [ ] 日文 UI 通过动态字体、窄屏、长文案、按钮宽度、换行回归。
- [ ] App Store Connect 日本区名称、副标题、描述、关键词、隐私 URL、客服 URL、审核说明完成。
- [ ] 日文法务和处罚通知经过人工确认并记录版本。

## 2. 账号状态、处罚与封禁体系

### 2.1 目标

账号需要支持被举报后审核确认封禁，也支持管理员手动封禁，并支持封禁一定天数。封禁体系必须能覆盖成熟商业产品常见场景：临时限制、临时封禁、永久封禁、申诉、恢复、复发加重、审计和通知。

### 2.2 账号状态模型

建议将账号基础状态与处罚状态拆开：

- 账号基础状态：
  - `active`：正常。
  - `pending_deletion`：删除申请处理中。
  - `deleted`：已删除/匿名化。
  - `disabled`：系统禁用，不一定是违规处罚。
- 处罚状态：
  - `none`：无处罚。
  - `restricted`：部分功能限制。
  - `suspended`：临时封禁，到期自动解除。
  - `banned`：永久封禁，需人工恢复。

处罚不应只用一个 `isActive=false` 表示，否则无法区分删除、封禁、风控冻结、法务保全。

### 2.3 处罚类型

- 警告：不限制功能，但通知用户并记录一次违规。
- 内容级处置：删除/隐藏/降权某条内容。
- 功能限制：
  - 禁止发帖。
  - 禁止评论。
  - 禁止私信/群聊。
  - 禁止上传图片/视频/音频。
  - 禁止创建活动/DJ/Set/内容投稿。
  - 禁止位置共享。
  - 禁止修改头像/昵称/简介。
- 临时封禁：
  - 常用时长：1 天、3 天、7 天、14 天、30 天、90 天、自定义日期。
  - 到期自动恢复。
  - 可提前人工解除。
- 永久封禁：
  - 仅管理员/超级管理员可操作。
  - 必须填写原因、证据、内部备注。
  - 支持申诉恢复。
- 风控冻结：
  - 可用于异常登录、盗号、支付争议、法律请求。
  - 与违规封禁分开展示和统计。

### 2.4 处罚原因码

原因码必须稳定，展示文案三语本地化：

- `spam`：垃圾信息/刷屏。
- `harassment`：骚扰/辱骂/霸凌。
- `hate_or_discrimination`：仇恨或歧视。
- `sexual_content`：色情或露骨内容。
- `violence_or_threat`：暴力威胁。
- `illegal_activity`：违法活动。
- `impersonation`：冒充他人/机构。
- `privacy_violation`：泄露隐私。
- `copyright`：版权侵权。
- `scam_or_fraud`：诈骗/钓鱼。
- `minor_safety`：未成年人安全。
- `platform_abuse`：绕过限制、恶意注册、自动化滥用。
- `other`：其他。

每个原因码需要：

- 用户可见简短说明。
- 管理后台详细说明。
- 默认处罚建议。
- 是否需要强制下架内容。
- 是否需要升级给高级审核员。

### 2.5 举报到封禁的完整链路

1. 用户举报账号或内容。
2. 系统创建 `ContentReport`，去重同一用户对同一对象的重复举报。
3. 系统聚合同一目标的多条举报，形成审核案件 `ModerationCase`。
4. 审核员查看目标、历史举报、历史处罚、关联内容、聊天上下文、证据截图/媒体。
5. 审核员选择处理：
   - 无违规：驳回举报。
   - 内容违规：下架内容、警告作者。
   - 用户违规：限制功能、临时封禁、永久封禁。
   - 需要升级：转高级审核。
6. 系统写处罚记录 `AccountEnforcement`。
7. 系统执行处罚：
   - 撤销/限制对应功能 token。
   - 隐藏或冻结相关内容。
   - 停止推送/私信能力。
   - 清理推荐流曝光。
8. 通知用户：
   - 被处罚用户收到处罚原因、范围、到期时间、申诉入口。
   - 举报人收到“已处理/未发现违规”的通用结果，不泄露被举报人隐私。
9. 到期自动解除临时封禁。
10. 用户可申诉，审核员可维持、减轻、解除、改判。
11. 所有操作进入审计和报表。

### 2.6 手动封禁完整链路

- 管理后台用户详情页提供“处罚账号”入口。
- 操作前展示用户风险画像：
  - 注册时间、登录设备、历史举报、历史处罚、发帖/消息频率、近期内容。
- 管理员选择处罚类型、范围、时长、原因码、用户可见说明、内部备注。
- 高风险操作要求二次确认：
  - 永久封禁。
  - 超过 30 天封禁。
  - 批量封禁。
  - 封禁认证用户/高权限用户。
- 可配置四眼审核：一个管理员发起，另一个管理员批准后生效。
- 处罚生效后所有端实时或下次请求感知。
- 管理员可延长、缩短、撤销处罚，但必须保留历史。

### 2.7 被封禁用户体验

- 登录前：
  - 临时封禁：展示封禁原因、到期时间、申诉入口、客服入口。
  - 永久封禁：展示永久封禁说明、申诉入口。
  - 部分限制：允许登录，但在受限操作时展示限制说明。
- 登录后：
  - 首页可访问有限信息，例如设置、申诉、数据导出、账号删除、法务页面。
  - 禁止绕过处罚：不能发帖、私信、改资料等。
- 所有封禁提示三语。
- 不向其他用户直接展示“该用户被封禁”的敏感原因，只显示“账号不可用”或隐藏内容。

### 2.8 申诉体系

- 用户可对每个处罚提交申诉。
- 申诉字段：
  - 处罚 ID。
  - 申诉理由。
  - 附件/截图。
  - 联系邮箱。
- 申诉状态：
  - `submitted`
  - `under_review`
  - `need_more_info`
  - `accepted`
  - `rejected`
  - `closed`
- 申诉处理结果：
  - 维持原处罚。
  - 解除处罚。
  - 缩短处罚。
  - 改为功能限制。
  - 永久封禁改为临时封禁。
- 同一处罚支持限制申诉次数，防止滥用，但必须保留客服入口。
- 申诉处理必须有管理员备注和通知。

### 2.9 数据模型建议

- `account_enforcements`
  - `id`
  - `user_id`
  - `status`
  - `type`
  - `scope`
  - `reason_code`
  - `user_message_i18n`
  - `internal_note`
  - `evidence`
  - `starts_at`
  - `ends_at`
  - `created_by`
  - `created_from_report_id`
  - `created_from_case_id`
  - `revoked_at`
  - `revoked_by`
  - `revocation_reason`
- `moderation_cases`
  - 聚合举报、目标对象、优先级、分配审核员、SLA、状态。
- `moderation_case_events`
  - 每次状态变化、备注、证据、处理动作。
- `enforcement_appeals`
  - 申诉文本、附件、处理结果。
- `user_risk_profiles`
  - 可选，聚合统计，不直接替代原始审计。

### 2.10 API 需求

- 用户端：
  - `GET /v1/account/status`
  - `GET /v1/account/enforcements`
  - `POST /v1/account/enforcements/:id/appeal`
  - `GET /v1/account/appeals`
- 管理端：
  - `GET /v1/admin/moderation/cases`
  - `GET /v1/admin/moderation/cases/:id`
  - `POST /v1/admin/moderation/cases/:id/decision`
  - `POST /v1/admin/users/:id/enforcements`
  - `PATCH /v1/admin/enforcements/:id`
  - `POST /v1/admin/enforcements/:id/revoke`
  - `GET /v1/admin/enforcement-appeals`
  - `POST /v1/admin/enforcement-appeals/:id/decision`
- 系统任务：
  - 到期自动解除临时处罚。
  - 到期前/到期后通知。
  - 异常失败重试。

### 2.11 验收标准

- 用户被封禁后无法通过 refresh token、旧 access token、IM token 继续使用受限能力。
- 临时封禁到期自动恢复，无需用户重装/重新注册。
- 管理员可手动封禁 1/3/7/14/30/90 天和自定义日期。
- 所有封禁操作都有审计、通知、申诉入口。
- 被举报内容可从举报案件跳转查看，上下文完整。
- 举报人和被举报人收到不同且合规的通知。
- 后台可按原因、状态、时长、处理人、SLA 筛选和导出。

### 2.12 商用交付 checklist

- [ ] 数据层区分账号基础状态、处罚状态、删除状态、风控冻结状态。
- [ ] 支持警告、内容处置、功能限制、临时封禁、永久封禁、风控冻结。
- [ ] 支持 1/3/7/14/30/90 天和自定义结束时间，临时处罚到期自动解除。
- [ ] 举报审核确认后可发起账号处罚，并关联原举报、案件、证据和审核员。
- [ ] 管理员可手动处罚账号，高风险处罚支持二次确认和四眼审核配置。
- [ ] 所有处罚有 reason code、用户可见三语说明、内部备注、证据、审计。
- [ ] 受限能力在 API、App、IM、推送、推荐、内容发布链路统一拦截。
- [ ] 被处罚用户可查看处罚范围、原因、到期时间、申诉入口、客服入口。
- [ ] 举报人收到处理结果通知，但不泄露被举报人隐私或处罚细节。
- [ ] 申诉支持提交、补充材料、审核、维持/减轻/解除/改判、通知和审计。
- [ ] 管理后台可查询、筛选、导出、撤销、延长、缩短、恢复处罚。
- [ ] 后台报表包含待审案件、超时案件、处罚量、申诉通过率、重复违规率。

## 3. UGC 举报、拉黑、内容审核

### 3.1 覆盖对象

- 用户资料：头像、昵称、简介、位置、外链。
- Feed 帖子：文字、图片、视频、位置。
- 帖子评论和回复。
- 活动 live comment。
- 私信、群聊消息、图片、视频、语音、文件、自定义卡片。
- DJ、Set、Tracklist、资讯、学习内容、活动投稿。
- 小队资料、公告、成员昵称、相册。

### 3.2 用户端举报

- 所有对象的更多菜单必须有举报入口。
- 举报原因三语展示，原因码统一。
- 支持补充说明、截图/附件、是否同时拉黑用户。
- 提交后展示处理说明和客服入口。
- 防重复：同一用户重复举报同一对象时更新补充说明，不创建无限重复。
- 用户可在设置中查看“我的举报记录”的处理状态。

### 3.3 拉黑

- 用户资料页、聊天设置页、举报成功页均可拉黑。
- 拉黑后：
  - 双方不能发起私信。
  - 对方内容从 Feed/评论/搜索推荐中隐藏或降权。
  - 对方不能邀请我进小队。
  - 现有会话隐藏或只读。
  - 小队内共同成员场景要有例外规则：可见必要系统消息，但不能私聊骚扰。
- 支持解除拉黑。
- 拉黑列表可在隐私设置中查看和管理。

### 3.4 审核后台

- 队列视图：按优先级、原因、对象类型、举报量、时间、SLA 筛选。
- 详情视图：对象预览、作者、举报人列表、上下文、历史处理、相似案件。
- 处理动作：
  - 驳回。
  - 下架内容。
  - 恢复内容。
  - 警告用户。
  - 限制功能。
  - 临时/永久封禁。
  - 升级审核。
- 批量处理：仅低风险同类垃圾内容可批量，必须审计。
- 处理模板：三语用户通知模板。

### 3.5 SLA 与告警

- 高危内容：未成年人安全、暴力威胁、违法、隐私泄露，需高优先级。
- 普通内容：垃圾信息、冒充、骚扰等。
- 超过 SLA 未处理需要后台告警。
- 每日输出待处理数量、超时数量、处罚数量、申诉通过率。

## 4. 账号删除

### 4.1 用户端

- 入口：设置 -> 账号安全 -> 删除账号。
- 展示影响：
  - 登录和会话失效。
  - 个人资料清除。
  - 公开内容处理策略。
  - 法定留存数据说明。
  - 删除后是否可恢复。
- 二次确认：输入确认文本或系统确认弹窗。
- 删除后退出登录并清理本地 token、IM、缓存、push token。
- 删除后仍可访问法务页面和客服入口。

### 4.2 服务端

- 删除请求幂等。
- 撤销所有 refresh token。
- 停用设备 push token。
- 停用或匿名化账号。
- 清理/匿名化：
  - 邮箱、手机号、头像、简介、位置。
  - 三方登录绑定。
  - IM 用户资料。
  - 分享码、二维码。
- 保留：
  - 审计日志。
  - 法定义务数据。
  - 已公开内容的匿名作者引用。
- 外部系统重试：
  - Tencent IM 账号删除/资料匿名化。
  - OSS 头像删除或脱敏。
  - 通知订阅清理。

### 4.3 后台与合规

- 后台可查看删除请求状态。
- 失败任务可重试。
- 用户可通过客服查询删除进度。
- 隐私政策说明删除范围和保存期限。

### 4.4 验收

- 删除后旧 access token/refresh token 全失效。
- 删除后不能通过手机号/邮箱直接恢复旧账号，除非设计明确恢复窗口。
- 用户个人资料不再公开展示。
- 管理后台有删除审计。

## 5. 法务、隐私与日本市场页面

### 5.1 页面

- `/legal/privacy`

- `/legal/contact`

- 使用目的。

- 未成年人。
- 联系方式。
- 账号删除。
- 推送、位置、UGC、聊天消息、媒体文件。

## 6. 权限、通知与位置安全

### 6.1 权限

- 权限弹窗前必须有软提示，说明用途和用户收益。
- 用户拒绝后保留替代路径：
  - 定位拒绝：手动输入地址。
  - 相机拒绝：从相册选择。
  - 麦克风拒绝：文字消息。
  - 推送拒绝：站内通知。
- 设置页提供权限状态说明和跳转系统设置。

### 6.2 推送

- 不在首次启动立即请求系统推送。
- 触发点：
  - 用户关注活动/DJ 后。
  - 用户进入通知设置。
  - 用户开启消息提醒。
- 通知偏好分类：
  - 私信/群聊。
  - 被关注/点赞/评论。
  - 活动提醒。
  - DJ/厂牌更新。
  - 审核/处罚/申诉。
  - 营销通知。
- 用户可关闭任意分类。

### 6.3 位置共享

- 加入 Squad 位置共享前说明：
  - 谁可见。
  - 更新频率。
  - 保存多久。
  - 如何停止。
- 活动结束自动停止位置上传。
- 用户可删除历史轨迹或提交删除请求。
- 后台可审计位置上传异常。

## 7. 隐私标签与 Privacy Manifest

### 7.1 数据地图

必须建立字段级数据地图：

- 字段名。
- 数据类型。
- 来源。
- 用途。
- 是否关联用户。
- 是否用于追踪。
- 是否共享给第三方。
- 保存期限。
- 删除策略。
- App Privacy 对应类型。

### 7.2 Manifest

- 主 App、Widget、Notification Service 都需要 `PrivacyInfo.xcprivacy`。
- 声明 Required Reason API：
  - UserDefaults。
  - 文件时间/元数据。
  - 其他扫描发现的 API。
- 第三方 SDK manifest 汇总进 Archive 隐私报告。

### 7.3 验收

- Xcode Archive 隐私报告可生成。
- App Store Connect 隐私标签与数据地图一致。
- 新增 SDK 或数据字段必须更新数据地图和隐私标签。

## 8. 商业化、IAP、虚拟资产

### 8.1 商业模式决策

- 票务外链。
- 自营会员。
- 数字装扮。
- 积分/点数。
- 投稿加速/推广。
- 虚拟活动权益。

每种模式必须判断：

- 是否必须走 Apple IAP。
- 是否触发特商法表示。
- 是否触发资金结算法/前払式支払手段。
- 是否需要退款/取消/恢复购买。
- 是否需要概率披露。

### 8.2 虚拟资产

- 定义获得方式：默认、活动奖励、运营发放、购买、兑换。
- 定义权利：不可转让、不可提现、不可兑换现金、是否过期。
- 购买类资产：
  - StoreKit 商品配置。
  - 购买凭证校验。
  - 恢复购买。
  - 退款通知处理。
  - 家庭共享判断。
- 随机类资产：
  - 概率展示。
  - 抽取记录。
  - 保底规则。
  - 未成年人限制。

## 9. 年龄分级与未成年人保护

- 注册或关键功能前采集出生年份/年龄段。
- 未成年限制：
  - 限制陌生人私信。
  - 限制位置共享。
  - 限制夜间活动票务跳转。
  - 限制成人内容曝光。
- 家长/监护人联系通道。
- 高危举报原因 `minor_safety` 优先级最高。
- App Store 年龄问卷按 UGC、聊天、位置共享、活动/酒精场景如实填写。

## 10. 知识产权与版权投诉

- 用户上传或提交音乐/视频链接时确认拥有权利或链接来源合法。
- 第三方音乐服务只做授权 API 或外链，不下载/转码非授权内容。
- 版权投诉页面提供：
  - 权利人信息。
  - 被投诉内容 URL/ID。
  - 权利证明。
  - 联系方式。
  - 反通知流程。
- 后台可处理版权投诉：
  - 临时下架。
  - 恢复。
  - 处罚重复侵权用户。
  - 记录证据和通知双方。
